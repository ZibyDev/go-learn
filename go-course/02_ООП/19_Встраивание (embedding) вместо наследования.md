# Урок 19 — Встраивание (embedding) вместо наследования

**Блок:** [[Go — План курса#📁 Блок 02 — ООП по-голански (~50 ч)|02 ООП по-голански]] **Время:** 4–5 часов **Теги:** #go/ооп #go/встраивание #go/композиция

---

## 📌 Теория

### 19.1 В Go нет наследования

Если ты приходишь из C#/Java/Kotlin, ты привык к наследованию: `class Dog extends Animal`. В Go этого **нет**. Совсем. Нет `extends`, нет базовых классов, нет иерархий.

Вместо наследования Go предлагает **встраивание (embedding)** — механизм композиции, который ты уже видел мельком в уроке 15 (анонимные поля). Сейчас разберём его полноценно.

> 💡 Это философский выбор Go: _«композиция вместо наследования»_. Идея не нова (её советуют и в ООП-языках), но Go делает её единственным путём. Наследование часто создаёт хрупкие иерархии; композиция гибче.

---

### 19.2 Встраивание структур — напоминание

Встраивание — это включение типа в структуру **без имени поля**, только тип:

```go
type Animal struct {
    Name string
}

type Dog struct {
    Animal   // встроено (анонимное поле) — НЕ "Animal Animal"
    Breed string
}
```

Поля встроенного типа **поднимаются** на верхний уровень (промоушен):

```go
d := Dog{
    Animal: Animal{Name: "Rex"},
    Breed:  "Labrador",
}

fmt.Println(d.Name)         // Rex — напрямую, как будто поле Dog
fmt.Println(d.Animal.Name)  // Rex — но можно и через полное имя
```

---

### 19.3 Промоушен методов — главное отличие от полей

Встраивание поднимает не только поля, но и **методы**. Это ключевая фишка: встроенный тип «делится» своими методами с внешним.

```go
type Animal struct {
    Name string
}

func (a Animal) Speak() string {
    return a.Name + " издаёт звук"
}

type Dog struct {
    Animal
    Breed string
}

func main() {
    d := Dog{Animal: Animal{Name: "Rex"}, Breed: "Lab"}
    fmt.Println(d.Speak())  // "Rex издаёт звук" — метод Animal доступен у Dog!
}
```

> 💡 Вот это похоже на наследование: `Dog` «получил» метод `Speak()` от `Animal`. Но механизм другой — это не наследование, а **делегирование**: `d.Speak()` на самом деле вызывает `d.Animal.Speak()`. Go просто избавляет от написания `d.Animal.`.

---

### 19.4 Переопределение методов

Внешний тип может объявить метод с тем же именем — он «перекроет» встроенный (метод внешнего типа приоритетнее):

```go
type Animal struct{ Name string }

func (a Animal) Speak() string {
    return a.Name + " издаёт звук"
}

type Dog struct {
    Animal
}

// Свой Speak у Dog — перекрывает Animal.Speak
func (d Dog) Speak() string {
    return d.Name + " говорит Гав!"
}

func main() {
    d := Dog{Animal: Animal{Name: "Rex"}}
    fmt.Println(d.Speak())         // "Rex говорит Гав!" — метод Dog
    fmt.Println(d.Animal.Speak())  // "Rex издаёт звук" — метод Animal явно
}
```

> 💡 Это похоже на override, но без `virtual`/`override`. Метод внешнего типа имеет приоритет. При этом ты всё ещё можешь вызвать встроенный метод явно через `d.Animal.Speak()` — как `super.speak()` в ООП-языках.

---

### 19.5 Встраивание и интерфейсы — мощная комбинация

Встроенные методы засчитываются при проверке реализации интерфейса. Если встроенный тип реализует интерфейс — внешний тип тоже его реализует:

```go
type Speaker interface {
    Speak() string
}

type Animal struct{ Name string }
func (a Animal) Speak() string { return a.Name + " говорит" }

type Dog struct {
    Animal   // встроили Animal, у которого есть Speak()
    Breed string
}

func main() {
    var s Speaker = Dog{Animal: Animal{Name: "Rex"}}  // Dog реализует Speaker!
    fmt.Println(s.Speak())  // через промоушен метода Animal
}
```

> 💡 Это очень практично: реализуй интерфейс один раз во встраиваемом типе, и все типы, которые его встроят, автоматически получат реализацию. Так в стандартной библиотеке часто встраивают `sync.Mutex`, `io.Reader` и т.д.

---

### 19.6 Встраивание интерфейсов в интерфейсы

Интерфейсы тоже можно встраивать — друг в друга. Так собирают большие интерфейсы из маленьких:

```go
type Reader interface {
    Read(p []byte) (int, error)
}

type Writer interface {
    Write(p []byte) (int, error)
}

// ReadWriter = Reader + Writer
type ReadWriter interface {
    Reader
    Writer
}
```

`ReadWriter` требует **оба** метода: `Read` и `Write`. Это реальный пример из пакета `io` (`io.ReadWriter`).

> 💡 Идиома Go: маленькие интерфейсы комбинируются в большие через встраивание. `io.ReadWriteCloser` = `Reader` + `Writer` + `Closer`. Гибко и переиспользуемо.

---

### 19.7 Встраивание указателя

Можно встроить не значение, а указатель на тип:

```go
type Engine struct {
    Power int
}

func (e *Engine) Start() string {
    return fmt.Sprintf("двигатель %d л.с. запущен", e.Power)
}

type Car struct {
    *Engine   // встроен указатель
    Model string
}

func main() {
    car := Car{
        Engine: &Engine{Power: 200},
        Model:  "Tesla",
    }
    fmt.Println(car.Start())  // методы Engine доступны
}
```

> 💡 Встраивание указателя полезно, когда встроенный объект большой (не копировать) или общий для нескольких структур. Но осторожно: если указатель `nil`, вызов промоутнутого метода вызовет панику.

---

### 19.8 Множественное встраивание

Можно встроить несколько типов сразу:

```go
type Logger struct{}
func (l Logger) Log(msg string) { fmt.Println("LOG:", msg) }

type Validator struct{}
func (v Validator) Validate() bool { return true }

type Service struct {
    Logger
    Validator
    Name string
}

func main() {
    s := Service{Name: "auth"}
    s.Log("запуск")        // от Logger
    fmt.Println(s.Validate())  // от Validator
}
```

> ⚠️ **Конфликт имён:** если два встроенных типа имеют метод/поле с одинаковым именем, обращение к нему **без уточнения** — ошибка компиляции (неоднозначность). Нужно явно: `s.Logger.X` или `s.Validator.X`. Внешний тип может разрешить конфликт, объявив свой метод с этим именем.

---

### 19.9 Почему композиция, а не наследование

Сравнение подходов:

**Наследование (ООП-языки):**

- «Dog ЕСТЬ Animal» (is-a)
- Жёсткая иерархия, хрупкая при изменениях
- Проблема «ромба» при множественном наследовании
- Глубокие деревья классов сложно поддерживать

**Композиция (Go):**

- «Dog ИМЕЕТ Animal» (has-a) — встраивание
- Гибкая сборка из частей
- Нет иерархий — плоская структура
- Легко комбинировать поведение

> 💡 Практическое правило Go: моделируй через «что объект УМЕЕТ» (интерфейсы) и «из чего СОСТОИТ» (встраивание), а не «чем он ЯВЛЯЕТСЯ» (наследование). Это даёт более гибкий и понятный код.

## 💡 Примеры

### Пример 1 — Промоушен полей и методов

```go
package main

import "fmt"

type Base struct {
    ID int
}

func (b Base) Describe() string {
    return fmt.Sprintf("ID=%d", b.ID)
}

type User struct {
    Base
    Name string
}

func main() {
    u := User{Base: Base{ID: 1}, Name: "Alice"}
    fmt.Println(u.ID)         // 1 — поле промоутнуто
    fmt.Println(u.Describe()) // ID=1 — метод промоутнут
}
```

### Пример 2 — Переопределение метода

```go
package main

import "fmt"

type Shape struct{ Name string }

func (s Shape) Area() float64 { return 0 }
func (s Shape) Info() string  { return s.Name + ": площадь " }

type Circle struct {
    Shape
    Radius float64
}

func (c Circle) Area() float64 { return 3.14159 * c.Radius * c.Radius }

func main() {
    c := Circle{Shape: Shape{Name: "круг"}, Radius: 5}
    fmt.Println(c.Area())  // 78.53 — метод Circle перекрыл Shape
}
```

### Пример 3 — Встраивание для реализации интерфейса

```go
package main

import "fmt"

type Greeter interface {
    Greet() string
}

type BaseGreeter struct{ Lang string }

func (b BaseGreeter) Greet() string {
    if b.Lang == "ru" {
        return "Привет"
    }
    return "Hello"
}

type Robot struct {
    BaseGreeter
    Model string
}

func main() {
    var g Greeter = Robot{BaseGreeter: BaseGreeter{Lang: "ru"}, Model: "R2D2"}
    fmt.Println(g.Greet())  // Привет — Robot реализует Greeter через встраивание
}
```

### Пример 4 — Множественное встраивание

```go
package main

import "fmt"

type Walker struct{}
func (w Walker) Move() string { return "идёт" }

type Swimmer struct{}
func (s Swimmer) Swim() string { return "плывёт" }

type Amphibian struct {
    Walker
    Swimmer
    Name string
}

func main() {
    a := Amphibian{Name: "лягушка"}
    fmt.Printf("%s %s и %s\n", a.Name, a.Move(), a.Swim())
    // лягушка идёт и плывёт
}
```

## ⚠️ Подводные камни

### 1. Встраивание ≠ наследование

```go
// Dog НЕ является подтипом Animal
// var a Animal = Dog{}  // ❌ это не работает как наследование
// Dog просто содержит Animal и делегирует ему
```

### 2. Конфликт имён при множественном встраивании

```go
type A struct{}
func (A) Do() {}
type B struct{}
func (B) Do() {}
type C struct{ A; B }
// c.Do()  // ❌ неоднозначность — нужно c.A.Do() или c.B.Do()
```

### 3. Анонимное поле = тип без имени

```go
type Dog struct {
    Animal Animal  // ❌ это ОБЫЧНОЕ поле с именем Animal, НЕ встраивание
    Animal         // ✅ встраивание (только тип)
}
```

### 4. nil указатель при встраивании указателя

```go
type Car struct{ *Engine }
c := Car{}       // Engine == nil
// c.Start()     // panic: nil pointer
```

### 5. Промоушен не работает «вверх»

```go
// Встроенный тип не знает о внешнем
func (a Animal) Speak() string {
    // здесь НЕТ доступа к полям Dog — Animal не знает, что он встроен
    return a.Name
}
```

### Задача 4 — Композиция поведения

### Задача 5 — Встраивание интерфейсов

## 📋 Краткое резюме

- В Go **нет наследования** — вместо него встраивание (embedding) и композиция.
- Встраивание — анонимное поле (только тип, без имени): `type Dog struct { Animal }`.
- Промоутятся и **поля**, и **методы** встроенного типа — доступны напрямую у внешнего.
- Механизм — делегирование: `d.Speak()` = `d.Animal.Speak()`, Go убирает посредника.
- Внешний тип может **переопределить** метод (его версия приоритетнее); оригинал доступен через `d.Animal.Method()`.
- Если встроенный тип реализует интерфейс — внешний тип тоже его реализует (промоушен методов засчитывается).
- Интерфейсы встраиваются друг в друга: большие = композиция маленьких (`io.ReadWriter`).
- Можно встраивать указатель (`*Engine`) — но осторожно с nil.
- Множественное встраивание возможно; конфликт имён → ошибка, разрешается уточнением или своим методом.
- Философия: моделируй через «УМЕЕТ» (интерфейсы) и «СОСТОИТ ИЗ» (встраивание), а не «ЯВЛЯЕТСЯ» (наследование).
