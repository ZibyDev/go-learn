# Урок 17 — Интерфейсы: неявная реализация, устройство

**Блок:** [[Go — План курса#📁 Блок 02 — ООП по-голански (~40 ч)|02 ООП по-голански]] **Время:** 5–6 часов **Теги:** #go/ооп #go/интерфейсы

## 📌 Теория

### 17.1 Что такое интерфейс

Интерфейс — это **набор сигнатур методов**. Тип, у которого есть все методы интерфейса, автоматически «реализует» его. Интерфейс описывает **поведение** («что умеет делать»), а не данные.

```go
type Shape interface {
    Area() float64
    Perimeter() float64
}
```

Любой тип с методами `Area() float64` и `Perimeter() float64` считается `Shape` — **автоматически**, без объявления.

---

### 17.2 Главная фишка Go — неявная реализация

Это самое важное отличие от Java/C#/Kotlin. Там пишут явно:

```java
// Java
class Circle implements Shape { ... }  // явное "implements"
```

В Go **нет** ключевого слова `implements`. Тип реализует интерфейс просто тем, что у него есть нужные методы:

```go
type Shape interface {
    Area() float64
}

type Circle struct {
    Radius float64
}

// Просто объявляем метод Area — и Circle УЖЕ реализует Shape
func (c Circle) Area() float64 {
    return 3.14159 * c.Radius * c.Radius
}

func main() {
    var s Shape = Circle{Radius: 5}  // работает! Circle реализует Shape
    fmt.Println(s.Area())            // 78.53975
}
```

> 💡 Это называется **duck typing на этапе компиляции**: «если что-то крякает как утка — это утка». Тип не знает, что реализует интерфейс. Связь устанавливается автоматически компилятором. Это даёт слабую связанность: интерфейс можно объявить **после** типа, в другом пакете.

---

### 17.3 Зачем нужны интерфейсы

Интерфейсы позволяют писать код, работающий с **любым** типом, обладающим нужным поведением:

```go
type Shape interface {
    Area() float64
}

type Circle struct{ Radius float64 }
type Rectangle struct{ Width, Height float64 }

func (c Circle) Area() float64    { return 3.14159 * c.Radius * c.Radius }
func (r Rectangle) Area() float64 { return r.Width * r.Height }

// Функция работает с ЛЮБОЙ фигурой
func printArea(s Shape) {
    fmt.Printf("Площадь: %.2f\n", s.Area())
}

func main() {
    printArea(Circle{Radius: 5})         // работает
    printArea(Rectangle{Width: 3, Height: 4})  // тоже работает
}
```

> 💡 Без интерфейсов пришлось бы писать `printCircleArea`, `printRectangleArea` и т.д. Интерфейс убирает дублирование: одна функция на все типы с нужным поведением. Это полиморфизм по-голангски.

---

### 17.4 Интерфейс как значение

Переменная интерфейсного типа может хранить значение **любого** типа, реализующего интерфейс:

```go
var s Shape

s = Circle{Radius: 5}      // храним Circle
fmt.Println(s.Area())

s = Rectangle{Width: 2, Height: 3}  // теперь Rectangle
fmt.Println(s.Area())
```

Под капотом значение интерфейса — это пара `(тип, значение)`:

- какой конкретный тип сейчас лежит
- само значение этого типа

> 💡 Это позволяет хранить разнородные объекты в одном срезе `[]Shape` — там могут быть и круги, и прямоугольники одновременно.

---

### 17.5 Срез интерфейсов — полиморфная коллекция

```go
shapes := []Shape{
    Circle{Radius: 5},
    Rectangle{Width: 3, Height: 4},
    Circle{Radius: 2},
}

total := 0.0
for _, s := range shapes {
    total += s.Area()   // у каждого свой Area()
}
fmt.Printf("Общая площадь: %.2f\n", total)
```

> 💡 Это и есть полиморфизм: один цикл вызывает `Area()`, и для каждого элемента выполняется **его** реализация. Какой именно метод вызвать, определяется в рантайме по конкретному типу.

---

### 17.6 Пустой интерфейс — interface{} / any

Интерфейс без методов реализует **любой** тип (у любого типа есть «ноль методов» из требуемых). Записывается `interface{}` или (с Go 1.18) псевдоним `any`:

```go
var x any        // any = interface{}
x = 42           // ок
x = "hello"      // ок
x = []int{1, 2}  // ок — любой тип
```

> 💡 `any` используется когда тип заранее неизвестен (например, `fmt.Println(...interface{})` принимает что угодно). Но злоупотреблять не стоит — теряется типобезопасность. Подробно работа с `any` (type assertion, type switch) — урок 18.

---

### 17.7 Маленькие интерфейсы — философия Go

В Go ценятся **маленькие** интерфейсы — часто из одного метода. Стандартная библиотека полна таких:

```go
// io.Reader — один метод
type Reader interface {
    Read(p []byte) (n int, err error)
}

// io.Writer — один метод
type Writer interface {
    Write(p []byte) (n int, err error)
}

// fmt.Stringer — один метод
type Stringer interface {
    String() string
}
```

> 💡 Идиома Go: _«чем больше интерфейс, тем слабее абстракция»_. Маленькие интерфейсы легче реализовать и комбинировать. Имя интерфейса из одного метода часто = метод + `er`: `Reader`, `Writer`, `Stringer`, `Closer`.

---

### 17.8 Stringer — ты его уже использовал

Помнишь метод `String()` из уроков 3 и 16? Он реализует интерфейс `fmt.Stringer`:

```go
type Stringer interface {
    String() string
}
```

Когда `fmt.Println` получает значение, он проверяет: реализует ли тип `Stringer`? Если да — вызывает `String()`. Вот почему твой метод `String()` автоматически подхватывался:

```go
type Color struct{ R, G, B uint8 }

func (c Color) String() string {
    return fmt.Sprintf("#%02X%02X%02X", c.R, c.G, c.B)
}

func main() {
    fmt.Println(Color{255, 0, 0})  // #FF0000 — fmt увидел Stringer
}
```

### 17.9 Интерфейсы и pointer receiver — важная тонкость

Вспомни method set из урока 16: у значения `T` только value-методы, у `*T` — и value, и pointer. Это влияет на реализацию интерфейсов:

```go
type Writer interface {
    Write(s string)
}

type File struct{ data string }

// pointer receiver
func (f *File) Write(s string) {
    f.data += s
}

func main() {
    var w Writer

    // w = File{}   // ❌ File НЕ реализует Writer (метод на *File)
    w = &File{}     // ✅ *File реализует Writer
    w.Write("hello")
}
```

> ⚠️ Если метод объявлен с **pointer receiver**, то интерфейс реализует только **указатель** `*T`, не значение `T`. Это частая ошибка новичков: «почему мой тип не реализует интерфейс?» — часто потому, что метод на `*T`, а в интерфейс кладут `T`.

---

### 17.10 Проверка реализации на этапе компиляции

Иногда нужно убедиться, что тип реализует интерфейс. Идиома — присваивание пустой переменной:

```go
type Shape interface {
    Area() float64
}

type Circle struct{ Radius float64 }
func (c Circle) Area() float64 { return 3.14 * c.Radius * c.Radius }

// Проверка на этапе компиляции: если Circle не реализует Shape — ошибка сборки
var _ Shape = Circle{}
var _ Shape = (*Circle)(nil)  // вариант для pointer receiver
```

> 💡 `var _ Shape = Circle{}` — выражение «убедись, что Circle это Shape». Если кто-то удалит метод `Area`, сборка сломается сразу, а не где-то в рантайме. Полезно в библиотеках.

---

### 17.11 nil-интерфейс и ловушка типизированного nil

Интерфейс равен `nil`, только если **и тип, и значение** равны nil:

```go
var s Shape
fmt.Println(s == nil)  // true — пустой интерфейс

// Ловушка: интерфейс с nil-значением, но НЕ-nil типом
var c *Circle = nil
var s2 Shape = c
fmt.Println(s2 == nil)  // false! тип (*Circle) есть, хоть значение nil
```

> ⚠️ Это знаменитая ловушка Go: интерфейс, в который положили nil-указатель конкретного типа, **не равен nil**. Потому что в паре (тип, значение) тип уже не nil. Это частый источник багов при возврате ошибок (урок 22). Пока просто знай, что такое бывает.

## 💡 Примеры

### Пример 1 — Полиморфизм через интерфейс

```go
package main

import (
    "fmt"
    "math"
)

type Shape interface {
    Area() float64
}

type Circle struct{ Radius float64 }
type Square struct{ Side float64 }

func (c Circle) Area() float64 { return math.Pi * c.Radius * c.Radius }
func (s Square) Area() float64 { return s.Side * s.Side }

func totalArea(shapes []Shape) float64 {
    total := 0.0
    for _, s := range shapes {
        total += s.Area()
    }
    return total
}

func main() {
    shapes := []Shape{
        Circle{Radius: 1},
        Square{Side: 2},
        Circle{Radius: 3},
    }
    fmt.Printf("%.2f\n", totalArea(shapes))
}
```

### Пример 2 — Несколько методов в интерфейсе

```go
package main

import "fmt"

type Animal interface {
    Sound() string
    Name() string
}

type Dog struct{ name string }
type Cat struct{ name string }

func (d Dog) Sound() string { return "Гав" }
func (d Dog) Name() string  { return d.name }
func (c Cat) Sound() string { return "Мяу" }
func (c Cat) Name() string  { return c.name }

func describe(a Animal) {
    fmt.Printf("%s говорит %s\n", a.Name(), a.Sound())
}

func main() {
    describe(Dog{name: "Рекс"})
    describe(Cat{name: "Барсик"})
}
```

### Пример 3 — Stringer

```go
package main

import "fmt"

type Money struct {
    Cents int
}

func (m Money) String() string {
    return fmt.Sprintf("%d.%02d руб", m.Cents/100, m.Cents%100)
}

func main() {
    price := Money{Cents: 1250}
    fmt.Println(price)  // 12.50 руб — Stringer сработал
}
```

### Пример 4 — Pointer receiver и интерфейс

```go
package main

import "fmt"

type Counter interface {
    Increment()
    Value() int
}

type counter struct{ n int }

func (c *counter) Increment() { c.n++ }
func (c *counter) Value() int { return c.n }

func main() {
    var c Counter = &counter{}  // нужен указатель — методы на *counter
    c.Increment()
    c.Increment()
    fmt.Println(c.Value())  // 2
}
```

## ⚠️ Подводные камни

### 1. Pointer receiver → интерфейс реализует только указатель

```go
func (f *File) Write() {}
var w Writer = File{}   // ❌ нужно &File{}
```

### 2. Забыл реализовать все методы интерфейса

```go
type Shape interface { Area() float64; Perimeter() float64 }
// Если у типа только Area — он НЕ Shape (не хватает Perimeter)
```

### 3. Типизированный nil не равен nil-интерфейсу

```go
var p *T = nil
var i Interface = p
// i != nil — хоть p и nil!
```

### 4. Слишком большой интерфейс

```go
// Плохо: интерфейс на 10 методов сложно реализовать
// Хорошо: маленькие интерфейсы, комбинируй при нужде
```

### 5. Интерфейс без значения паникует при вызове

```go
var s Shape       // nil
// s.Area()       // panic: nil pointer dereference
```

## 📋 Краткое резюме

- Интерфейс — набор сигнатур методов; описывает поведение, не данные.
- **Неявная реализация:** тип реализует интерфейс автоматически, если имеет все его методы. Нет `implements`.
- Это даёт слабую связанность: интерфейс можно объявить после типа и в другом пакете.
- Интерфейс-значение хранит пару (конкретный тип, значение); может держать любой реализующий тип.
- Срез интерфейсов `[]Shape` — полиморфная коллекция; один цикл, разные реализации метода.
- Пустой интерфейс `interface{}` / `any` реализует любой тип (ноль требуемых методов).
- Философия Go: маленькие интерфейсы (часто 1 метод): `Reader`, `Writer`, `Stringer`.
- `fmt.Stringer` (`String() string`) — `fmt` автоматически использует его при печати.
- Если метод на pointer receiver — интерфейс реализует только `*T`, не `T`.
- Проверка реализации на компиляции: `var _ Iface = Type{}`.
- Ловушка: интерфейс с nil-указателем конкретного типа **не равен** nil.
