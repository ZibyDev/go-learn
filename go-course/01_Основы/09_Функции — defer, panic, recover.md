## 📌 Теория

### 9.1 defer — отложенное выполнение

`defer` откладывает выполнение функции до момента, когда **окружающая функция завершится** (через return, panic или просто дойдя до конца).

```go
func main() {
    defer fmt.Println("это выполнится последним")
    fmt.Println("это первым")
    fmt.Println("это вторым")
}
// Вывод:
// это первым
// это вторым
// это выполнится последним
```

> 💡 Главное применение — **гарантированное освобождение ресурсов**: закрытие файлов, соединений, разблокировка мьютексов. Что бы ни случилось в функции — defer выполнится.

---

### 9.2 Зачем нужен defer

Сравни два подхода к закрытию файла:

```go
// ❌ Без defer — легко забыть закрыть, особенно при нескольких return
func readFile() error {
    f, err := os.Open("file.txt")
    if err != nil {
        return err
    }
    // ...много кода...
    if somethingWrong {
        f.Close()  // не забыть!
        return err
    }
    // ...ещё код...
    f.Close()  // и тут не забыть!
    return nil
}

// ✅ С defer — закроется в любом случае
func readFile() error {
    f, err := os.Open("file.txt")
    if err != nil {
        return err
    }
    defer f.Close()  // закроется при ЛЮБОМ выходе из функции

    // ...сколько угодно кода и return'ов...
    return nil
}
```

> 💡 Идиома Go: **открыл ресурс → сразу пиши `defer закрытие`**. Тогда точно не забудешь, и код читается линейно.

---

### 9.3 Порядок выполнения defer — LIFO (стек)

Несколько `defer` выполняются в **обратном** порядке (Last In, First Out):

```go
func main() {
    defer fmt.Println("1")
    defer fmt.Println("2")
    defer fmt.Println("3")
}
// Вывод:
// 3
// 2
// 1
```

> 💡 Это логично: если ты открыл ресурс A, потом B (зависящий от A), то закрывать нужно сначала B, потом A. LIFO как раз даёт правильный порядок.

---

### 9.4 Важная ловушка: аргументы defer вычисляются СРАЗУ

Аргументы отложенной функции вычисляются **в момент объявления defer**, а не в момент выполнения:

```go
func main() {
    x := 10
    defer fmt.Println("defer видит x =", x)  //x вычислен СЕЙЧАС = 10
    x = 20
    fmt.Println("в конце x =", x)
}
// Вывод:
// в конце x = 20
// defer видит x = 10   ← не 20!
```

Если нужно "позднее" значение — оберни в замыкание:

```go
func main() {
    x := 10
    defer func() {
        fmt.Println("defer видит x =", x)  // вычислится в момент выполнения = 20
    }()
    x = 20
}
// defer видит x = 20
```

> ⚠️ Это очень частый источник путаницы. Запомни: `defer f(x)` фиксирует `x` сразу, `defer func(){ ...x... }()` берёт `x` в момент выполнения.

---

### 9.5 defer может менять именованные возвраты

`defer` с замыканием может модифицировать именованные возвращаемые значения **после** `return`:

```go
func double() (result int) {
    defer func() {
        result *= 2  // меняем уже после return!
    }()
    return 5  // result = 5, потом defer удвоит → 10
}

func main() {
    fmt.Println(double())  // 10
}
```

Механика: `return 5` сначала присваивает `result = 5`, **затем** выполняются defer'ы (которые видят и могут менять `result`), и только потом функция реально возвращает значение.

> 💡 Это база для перехвата паники и преобразования её в ошибку (см. ниже). Связка с именованными возвратами из урока 8.

---

### 9.6 panic — аварийная остановка

`panic` — это механизм аварийного завершения. Когда вызывается `panic`, нормальное выполнение прекращается, выполняются все отложенные `defer`, и программа падает с сообщением и стеком вызовов.

```go
func main() {
    fmt.Println("до паники")
    panic("что-то пошло не так")
    fmt.Println("после паники")  // НЕ выполнится
}
// до паники
// panic: что-то пошло не так
// goroutine 1 [running]:
// ...стек вызовов...
// exit status 2
```

Паника возникает и автоматически — при ошибках рантайма:

```go
var s []int
_ = s[10]        // panic: index out of range

var m map[string]int
m["key"] = 1     // panic: assignment to entry in nil map

var p *int
_ = *p           // panic: nil pointer dereference

x := 0
_ = 10 / x       // panic: integer divide by zero
```

> ⚠️ **Философия Go:** `panic` — НЕ замена исключений. Для ожидаемых ошибок используй `error` (урок 8). `panic` — только для **действительно непредвиденных** ситуаций, когда продолжать невозможно (баг в программе, нарушение инварианта).

---

### 9.7 recover — перехват паники

`recover` останавливает панику и возвращает управление. Работает **только внутри `defer`**:

```go
func safeCall() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("перехватили панику:", r)
        }
    }()

    panic("ой!")
}

func main() {
    safeCall()
    fmt.Println("программа продолжает работать")  // выполнится!
}
// перехватили панику: ой!
// программа продолжает работать
```

> 💡 `recover()` вне `defer` вернёт `nil` и ничего не сделает. Он работает только в отложенной функции, потому что только там есть доступ к "падающей" горутине.

---

### 9.8 Идиома: преобразование паники в ошибку

Самое практичное применение `recover` — превратить панику в обычную `error` через именованный возврат:

```go
func safeProcess(data []int) (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("восстановление после паники: %v", r)
        }
    }()

    // потенциально опасный код
    process(data)
    return nil
}
```

Здесь сходятся три темы: именованный возврат (`err error`), defer-замыкание, и recover. Если внутри случится паника — функция вернёт её как обычную ошибку, и вызывающий код обработает её штатно.

> ⚠️ Это **не** значит, что нужно оборачивать всё подряд в recover. Идиома применяется на границах: например, в обработчике HTTP-запроса, чтобы паника в одном запросе не уронила весь сервер.

---

### 9.9 Когда что использовать

|Ситуация|Инструмент|
|---|---|
|Ожидаемая ошибка (файл не найден, неверный ввод)|`error` (возврат значения)|
|Освобождение ресурса|`defer`|
|Непредвиденный баг, продолжать нельзя|`panic`|
|Граница, где нельзя дать упасть всему (сервер)|`recover` в defer|

> 💡 Правило: в библиотечном коде паникуй только в крайних случаях, **возвращай ошибки**. Паника, вылетевшая из пакета, — почти всегда плохой дизайн API.

## 💡 Примеры

### Пример 1 — defer для пары "открыл/закрыл"

```go
package main

import "fmt"

func openResource(name string) {
    fmt.Println("открыли", name)
}
func closeResource(name string) {
    fmt.Println("закрыли", name)
}

func main() {
    openResource("файл")
    defer closeResource("файл")

    fmt.Println("работаем с файлом")
    // closeResource выполнится автоматически в конце
}
// открыли файл
// работаем с файлом
// закрыли файл
```

### Пример 2 — LIFO порядок и ловушка аргументов

```go
package main

import "fmt"

func main() {
    for i := 1; i <= 3; i++ {
        defer fmt.Println("defer аргумент:", i)  // i фиксируется сразу
    }
    fmt.Println("конец main")
}
// конец main
// defer аргумент: 3
// defer аргумент: 2
// defer аргумент: 1
```

### Пример 3 — recover превращает панику в ошибку

```go
package main

import "fmt"

func safeDivide(a, b int) (result int, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("ошибка деления: %v", r)
        }
    }()
    result = a / b  // при b==0 будет паника
    return result, nil
}

func main() {
    if r, err := safeDivide(10, 2); err != nil {
        fmt.Println("ошибка:", err)
    } else {
        fmt.Println("10/2 =", r)
    }

    if _, err := safeDivide(10, 0); err != nil {
        fmt.Println("ошибка:", err)  // перехваченная паника
    }

    fmt.Println("программа жива")
}
// 10/2 = 5
// ошибка: ошибка деления: runtime error: integer divide by zero
// программа жива
```

### Пример 4 — defer меняет именованный возврат

```go
package main

import "fmt"

func withCleanup() (status string) {
    defer func() {
        status = status + " (очищено)"
    }()
    return "готово"
}

func main() {
    fmt.Println(withCleanup())  // готово (очищено)
}
```

## ⚠️ Подводные камни

### 1. Аргументы defer вычисляются сразу

```go
x := 1
defer fmt.Println(x)  // зафиксирует 1
x = 2
// выведет 1, не 2
```

### 2. recover работает ТОЛЬКО в defer

```go
func bad() {
    if r := recover(); r != nil { }  // ❌ recover вне defer = всегда nil
    panic("упадёт")
}
```

### 3. defer в цикле копит вызовы до конца функции

```go
func leak() {
    for i := 0; i < 1000000; i++ {
        f, _ := os.Open("file")
        defer f.Close()  // ❌ накопится миллион defer'ов, файлы не закрываются по ходу!
    }
}
// Решение: вынести тело цикла в отдельную функцию, чтобы defer срабатывал на каждой итерации
```

### 4. panic не для обычных ошибок

```go
// ❌ Плохо
func getUser(id int) User {
    if id < 0 {
        panic("неверный id")  // это ожидаемая ситуация!
    }
}

// ✅ Хорошо
func getUser(id int) (User, error) {
    if id < 0 {
        return User{}, fmt.Errorf("неверный id: %d", id)
    }
}
```

### 5. Голый recover без проверки на nil

```go
defer func() {
    recover()  // молча проглатывает ВСЁ, включая то что не ожидал — плохо
}()
// Лучше: проверяй r != nil и логируй/оборачивай
```

## 📋 Краткое резюме

- `defer` откладывает вызов до конца функции; идиома «открыл ресурс → сразу defer закрытие».
- Несколько `defer` выполняются в обратном порядке (LIFO).
- Аргументы `defer f(x)` вычисляются **сразу**; для «позднего» значения — обернуть в замыкание.
- `defer` с замыканием может менять именованные возвраты после `return`.
- `panic` — аварийная остановка для непредвиденного; для ожидаемых ошибок — `error`.
- `recover` перехватывает панику, работает **только внутри** `defer`.
- Идиома: превратить панику в `error` через именованный возврат + defer + recover.
- В библиотечном коде возвращай ошибки, а не паникуй.
